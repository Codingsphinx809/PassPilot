# Design System

This document defines the visual language and UI standards for PassPilot.

The goal is to create a consistent, professional, and accessible experience across every screen.

---

# Design Philosophy

PassPilot should feel:

- Modern
- Professional
- Calm
- Encouraging
- Trustworthy

The interface should help users focus on learning, not on navigating the application.

---

# Core Principles

## Simplicity

Every screen should have one primary purpose.

Avoid unnecessary complexity.

---

## Consistency

Similar actions should always look and behave the same.

Users should never have to relearn the interface.

---

## Accessibility

Accessibility is a requirement.

Support:

- Dynamic text sizing
- High contrast
- Screen readers
- Large touch targets
- Color-independent indicators

---

---

# Spacing

Use an 8-point spacing system.

Examples:

- 8px
- 16px
- 24px
- 32px
- 40px
- 48px

Avoid arbitrary spacing values whenever possible.

---

# Border Radius

Small

8px

Medium

12px

Large

20px

Cards should have soft rounded corners.

---

# Shadows

Use subtle shadows.

Depth should communicate hierarchy—not decoration.

---

# Icons

Use one icon library throughout the application.

Recommended:

- Lucide
- Material Symbols

Icons should always support labels where appropriate.

---

# Buttons

Primary

Filled button.

Highest emphasis.

---

Secondary

Outlined button.

Medium emphasis.

---

Text Button

Lowest emphasis.

Used for optional actions.

---

# Cards

Cards should contain:

- Clear title
- Supporting content
- Appropriate spacing
- Optional actions

Avoid overcrowding.

---

# Forms

Forms should:

- Validate immediately where appropriate
- Clearly identify required fields
- Display helpful error messages
- Minimize typing whenever possible

---

# Navigation

Primary navigation should always be visible.

Users should always know:

- Where they are
- Where they can go
- What comes next

---

# Animations

Animations should:

- Feel smooth
- Be subtle
- Reinforce actions
- Never delay users

Examples:

- Button feedback
- Screen transitions
- Progress updates
- Achievement celebrations

---

# Empty States

Every empty state should:

- Explain why the page is empty
- Suggest the next action
- Encourage continued learning

---

# Loading States

Never leave blank screens.

Use:

- Skeleton loaders
- Progress indicators
- Activity animations

---

# Error States

Error messages should:

- Explain what happened
- Explain how to fix it
- Avoid technical jargon

---

# Component Library

The design system should eventually include reusable components for:

- Buttons
- Cards
- Progress bars
- Lesson cards
- Quiz cards
- Flashcards
- Navigation bars
- Dialogs
- Bottom sheets
- Input fields
- Charts
- Badges
- Achievement cards
- Avatars
- Loading indicators

---

# Guiding Principle

Every screen should feel like it belongs to the same product.

A learner should never notice the design system—but they should always benefit from its consistency.
